# ToDoList MVP 接口设计文档

## 一、文档目标

本文档用于定义 ToDoList 项目第一版（MVP）的后端接口设计。  
目标是让前端、后端、数据库、业务规则之间保持一致，方便后续直接进入开发阶段。

本版本围绕以下核心闭环展开：

**添加任务 → 完成任务 → 获得积分 → 兑换奖励 → 查看目标进度**

---

## 二、接口设计原则

第一版接口设计遵循以下原则：

1. **优先满足 MVP**
   - 只设计当前第一版真正需要的接口
   - 不提前引入过多高级接口

2. **接口语义清晰**
   - 路径尽量和资源一致
   - 操作尽量符合 REST 风格

3. **便于 Java Spring Boot 实现**
   - 接口结构尽量适合 Controller / Service / DTO 分层
   - 请求和响应结构尽量统一

4. **为后续扩展留空间**
   - 虽然第一版简单，但接口命名不要太死
   - 后续可以平滑扩展为更完整的项目

---

## 三、MVP 接口总览

第一版建议至少提供以下接口：

### 任务相关
- `GET /api/tasks`
- `POST /api/tasks`
- `PATCH /api/tasks/{id}/complete`
- `DELETE /api/tasks/{id}`

### 奖励相关
- `GET /api/rewards`
- `POST /api/rewards`
- `POST /api/rewards/{id}/redeem`

### 目标相关
- `GET /api/goals/current`
- `POST /api/goals`
- `PATCH /api/goals/{id}`

### 用户统计相关
- `GET /api/user-stats`

## 五、任务相关接口

### 5.1 获取任务列表

#### 接口说明
获取当前任务列表，用于首页展示任务。

#### 请求方式
`GET /api/tasks`

#### 可选查询参数

- `status`
  - 可选
  - 值：
    - `TODO`
    - `COMPLETED`

- `isLongTerm`
  - 可选
  - 值：
    - `true`
    - `false`

- `goalId`
  - 可选
  - 用于查询某个目标下的任务

- `timeSlot`
  - 可选
  - 值：
    - `MORNING`
    - `AFTERNOON`
    - `EVENING`

#### 请求示例

```http
GET /api/tasks?status=TODO&isLongTerm=false
```

#### 响应示例

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": "1",
      "title": "刷题30分钟",
      "description": "完成行测练习",
      "type": "ONCE",
      "status": "TODO",
      "points": 20,
      "isLongTerm": false,
      "goalId": null,
      "timeSlot": "MORNING",
      "createdAt": "2026-03-22T09:00:00",
      "updatedAt": "2026-03-22T09:00:00",
      "completedAt": null
    }
  ]
}
```

### 5.2 新增任务

#### 接口说明
创建一个新的任务，用于首页添加任务。

#### 请求方式
`POST /api/tasks`

#### 请求体

```json
{
  "title": "背单词",
  "description": "复习50个单词",
  "type": "REPEAT",
  "points": 10,
  "isLongTerm": true,
  "goalId": "1",
  "timeSlot": "EVENING"
}
```

#### 字段说明

- `title`
  - 必填
  - 任务名称

- `description`
  - 可选
  - 任务说明

- `type`
  - 必填
  - `ONCE` 或 `REPEAT`

- `points`
  - 必填
  - 非负整数

- `isLongTerm`
  - 必填
  - 是否长期任务

- `goalId`
  - 可选
  - 关联目标 ID

- `timeSlot`
  - 可选
  - `MORNING` / `AFTERNOON` / `EVENING`

#### 响应示例

```json
{
  "code": 200,
  "message": "任务创建成功",
  "data": {
    "id": "2",
    "title": "背单词",
    "description": "复习50个单词",
    "type": "REPEAT",
    "status": "TODO",
    "points": 10,
    "isLongTerm": true,
    "goalId": "1",
    "timeSlot": "EVENING",
    "createdAt": "2026-03-22T10:00:00",
    "updatedAt": "2026-03-22T10:00:00",
    "completedAt": null
  }
}
```

### 5.3 完成任务

#### 接口说明
将任务标记为完成，并增加对应积分。

#### 请求方式
`PATCH /api/tasks/{id}/complete`

#### 路径参数

- `id`
  - 必填
  - 任务 ID

#### 请求示例

```http
PATCH /api/tasks/2/complete
```

#### 业务规则

- 任务必须存在
- 已完成任务不能重复完成
- 完成成功后：
  - 任务状态改为 `COMPLETED`
  - 写入 `completedAt`
  - 用户积分增加
  - 用户完成任务数增加
- 如果任务关联目标，可同步更新目标进度
  - 第一版可以先做简单更新或不自动更新

#### 响应示例

```json
{
  "code": 200,
  "message": "任务完成成功",
  "data": {
    "taskId": "2",
    "taskStatus": "COMPLETED",
    "gainedPoints": 10,
    "totalPoints": 120,
    "completedAt": "2026-03-22T18:30:00"
  }
}
```

---

### 5.4 删除任务

#### 接口说明
删除某个任务。

#### 请求方式
`DELETE /api/tasks/{id}`

#### 路径参数

- `id`
  - 必填
  - 任务 ID

#### 请求示例

```http
DELETE /api/tasks/2
```

#### 业务规则

- 任务必须存在
- 第一版建议：
  - 删除任务不回滚已获得积分
  - 已完成任务删除后也不回退积分

#### 响应示例

```json
{
  "code": 200,
  "message": "任务删除成功",
  "data": null
}
```

### 6.3 兑换奖励

#### 接口说明
兑换指定奖励，并扣除对应积分。

#### 请求方式
`POST /api/rewards/{id}/redeem`

#### 路径参数

- `id`
  - 必填
  - 奖励 ID

#### 请求示例

```http
POST /api/rewards/2/redeem
```

#### 业务规则

- 奖励必须存在
- 奖励状态必须为 `ACTIVE`
- 当前用户积分必须足够
- 兑换成功后：
  - 扣除用户积分
- 第一版先不记录兑换历史

#### 响应示例

```json
{
  "code": 200,
  "message": "奖励兑换成功",
  "data": {
    "rewardId": "2",
    "rewardTitle": "玩一小时游戏",
    "costPoints": 80,
    "remainingPoints": 40
  }
}
```

#### 积分不足响应示例

```json
{
  "code": 400,
  "message": "积分不足，无法兑换",
  "data": null
}
```

## 十一、第一版核心业务校验建议

为了让接口更稳，建议后端先做好以下校验：

### 1. 创建任务时
- `title` 不能为空
- `points` 不能小于 0
- `type` 必须合法
- `timeSlot` 如果传入，必须合法

### 2. 完成任务时
- 任务必须存在
- 任务必须是未完成状态
- 已完成任务不可重复完成

### 3. 创建奖励时
- `title` 不能为空
- `costPoints` 不能小于 0

### 4. 兑换奖励时
- 奖励必须存在
- 奖励必须启用
- 积分必须足够

### 5. 更新目标时
- 目标必须存在
- `progress` 必须在 0 到 100 之间

---

## 十二、第一版建议的异常响应场景

建议至少覆盖这些常见异常：

- 参数错误
- 资源不存在
- 任务重复完成
- 积分不足
- 目标不存在
- 奖励不存在

### 示例：资源不存在

```json
{
  "code": 404,
  "message": "任务不存在",
  "data": null
}
```

### 示例：重复完成任务

```json
{
  "code": 400,
  "message": "任务已完成，不能重复提交",
  "data": null
}
```
