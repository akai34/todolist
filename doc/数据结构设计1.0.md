# ToDoList MVP 数据结构设计文档

## 一、文档目标

本文档用于定义 ToDoList 项目第一版（MVP）所需的核心数据结构。  
目标是让页面设计、后端接口、数据库表结构、前端类型定义尽量统一，减少后续返工。

本版本围绕以下核心闭环设计：

**添加任务 → 完成任务 → 获得积分 → 兑换奖励 → 查看长期目标进度**

---

## 二、设计原则

第一版数据结构设计遵循以下原则：

1. **先满足 MVP**
   - 只保留当前页面和核心逻辑真正需要的数据
   - 不提前引入过多复杂字段

2. **保留扩展空间**
   - 虽然第一版不做复杂功能，但字段命名和结构要方便以后扩展
   - 例如：长期任务、目标关联、重复任务等

3. **前后端易于理解**
   - 数据结构尽量直观
   - 字段名统一、语义清晰

4. **优先适合 Java 后端实现**
   - 结构设计要便于映射为 Java 实体类、DTO、数据库表

---

## 三、MVP 核心实体总览

第一版建议包含以下 4 个核心实体：

1. `Task`：任务
2. `Reward`：奖励
3. `Goal`：目标
4. `UserStats`：用户积分统计

如果后端要做成更标准的项目，后续还可以扩展：

- `RewardRedeemRecord`：奖励兑换记录
- `TaskCompleteRecord`：任务完成记录
- `User`：用户信息

但这些在第一版中可以先不强制加入。

---

## 四、Task（任务）

### 4.1 实体说明

`Task` 是整个系统最核心的数据对象。  
它用于承载首页中的任务列表、长期任务、任务完成、积分发放等逻辑。

任务在第一版中分为两种：

- 一次性任务
- 重复任务

同时，任务还可以进一步区分：

- 是否是长期任务
- 是否关联某个目标

---

### 4.2 字段设计

```ts
type Task = {
  id: string
  title: string
  description?: string
  type: 'ONCE' | 'REPEAT'
  status: 'TODO' | 'COMPLETED'
  points: number
  isLongTerm: boolean
  goalId?: string
  timeSlot?: 'MORNING' | 'AFTERNOON' | 'EVENING'
  displayOrder?: number
  createdAt: string
  updatedAt: string
  completedAt?: string
}
```

---

### 4.3 字段说明

- `id`
  - 任务唯一标识
  - 推荐使用字符串类型，方便前后端统一处理

- `title`
  - 任务名称
  - 必填
  - 用于任务列表主标题显示

- `description`
  - 任务说明
  - 可选
  - 用于补充任务内容

- `type`
  - 任务类型
  - 可选值：
    - `ONCE`：一次性任务
    - `REPEAT`：重复任务

- `status`
  - 当前任务状态
  - 可选值：
    - `TODO`：未完成
    - `COMPLETED`：已完成

- `points`
  - 完成该任务可获得的积分
  - 必须为非负整数

- `isLongTerm`
  - 是否为长期任务
  - `true` 表示该任务会出现在目标页 / 长期任务区
  - `false` 表示普通任务

- `goalId`
  - 关联目标 ID
  - 可选
  - 如果任务属于某个目标，则填入对应目标 ID

- `timeSlot`
  - 时间段标签
  - 可选
  - 第一版只做轻量分类，不做复杂排程
  - 可选值：
    - `MORNING`
    - `AFTERNOON`
    - `EVENING`

- `displayOrder`
  - 展示顺序
  - 可选
  - 用于控制任务在列表中的前后顺序

- `createdAt`
  - 创建时间

- `updatedAt`
  - 更新时间

- `completedAt`
  - 完成时间
  - 可选
  - 仅当任务完成时有值

---

### 4.4 第一版业务说明

#### 一次性任务

- 创建后默认是 `TODO`
- 用户完成后变为 `COMPLETED`
- 获得积分
- 后续可继续保留在已完成列表或隐藏

#### 重复任务

第一版先简化处理：

- 当天完成后状态为 `COMPLETED`
- 是否自动次日重置，第一版可以先不做自动逻辑
- 可先通过手动重置或简单规则实现

---

## 五、Reward（奖励）

### 5.1 实体说明

`Reward` 用于承载奖励页中的奖励列表与兑换逻辑。  
它和普通任务无关，但会和用户积分产生直接关系。

---

### 5.2 字段设计

```ts
type Reward = {
  id: string
  title: string
  description?: string
  costPoints: number
  status: 'ACTIVE' | 'INACTIVE'
  createdAt: string
  updatedAt: string
}
```

---

### 5.3 字段说明

- `id`
  - 奖励唯一标识

- `title`
  - 奖励名称
  - 例如：
    - 喝奶茶
    - 玩一小时游戏
    - 买一本书

- `description`
  - 奖励说明
  - 可选
  - 用于补充奖励内容

- `costPoints`
  - 兑换该奖励需要消耗的积分
  - 必须为非负整数

- `status`
  - 奖励状态
  - 可选值：
    - `ACTIVE`：启用中，可兑换
    - `INACTIVE`：停用，不可兑换

- `createdAt`
  - 创建时间

- `updatedAt`
  - 更新时间

---

### 5.4 第一版业务说明

- 奖励默认可兑换
- 若用户积分足够，则可兑换成功
- 兑换成功后用户积分减少
- 第一版不强制记录兑换历史



---

## 六、Goal（目标）

### 6.1 实体说明

`Goal` 用于承载目标页中的长期目标与进度展示逻辑。  
它是你当前项目中比较有特色的一部分，用来表达：

- 我正在为什么努力
- 我的长期进度怎么样

---

### 6.2 字段设计

```ts
type Goal = {
  id: string
  title: string
  description?: string
  progress: number
  status: 'IN_PROGRESS' | 'COMPLETED'
  startDate?: string
  endDate?: string
  createdAt: string
  updatedAt: string
}
```

---

### 6.3 字段说明

- `id`
  - 目标唯一标识

- `title`
  - 目标名称
  - 例如：
    - 3月统考冲刺
    - Java 后端项目上线
    - 春招准备计划

- `description`
  - 目标说明
  - 可选

- `progress`
  - 当前进度百分比
  - 推荐范围：`0 ~ 100`

- `status`
  - 目标状态
  - 可选值：
    - `IN_PROGRESS`
    - `COMPLETED`

- `startDate`
  - 目标开始日期
  - 可选

- `endDate`
  - 目标截止日期
  - 可选

- `createdAt`
  - 创建时间

- `updatedAt`
  - 更新时间

---

### 6.4 第一版业务说明

第一版的 `Goal` 建议做简单一点：

- 只展示一个当前目标
- 目标进度可以先手动维护
- 或者通过关联任务的完成情况进行非常简单的计算

第一版先不要做：

- 多目标复杂切换
- 自动复杂进度计算
- 目标树结构

---

## 七、UserStats（用户积分统计）

### 7.1 实体说明

`UserStats` 用于保存当前用户的一些全局统计信息。  
第一版最重要的是：

- 当前总积分

后续还可以扩展：

- 累计完成任务数
- 累计兑换奖励数
- 连续打卡天数

---

### 7.2 字段设计

```ts
type UserStats = {
  userId: string
  totalPoints: number
  totalCompletedTasks: number
  updatedAt: string
}
```

---

### 7.3 字段说明

- `userId`
  - 用户标识
  - 如果第一版暂时不做登录，可以先固定一个默认用户

- `totalPoints`
  - 当前可用总积分

- `totalCompletedTasks`
  - 累计完成任务数
  - 可选但建议保留
  - 后面可以用于成长展示

- `updatedAt`
  - 最近更新时间


---

## 八、实体关系说明

第一版核心实体之间的关系如下：

### 1. Task 与 Goal

- 一个任务可以关联一个目标
- 一个目标可以包含多个任务

即：

- `Goal 1 ---- N Task`

通过 `Task.goalId` 建立关联。

---

### 2. Reward 与 UserStats

- 奖励本身不直接存积分
- 兑换时读取 `UserStats.totalPoints`
- 判断是否满足 `Reward.costPoints`

---

### 3. Task 与 UserStats

- 完成任务后：
  - `Task.status` 变为 `COMPLETED`
  - `UserStats.totalPoints` 增加
  - `UserStats.totalCompletedTasks` 增加

---

## 九、推荐数据库表设计（第一版）

如果后端用 Java + Spring Boot + MySQL，第一版建议至少建这 4 张表：

1. `task`
2. `reward`
3. `goal`
4. `user_stats`

---

## 十、推荐表结构草案

### 10.1 task 表

```sql
CREATE TABLE task (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100) NOT NULL,
  description VARCHAR(255),
  type VARCHAR(20) NOT NULL,
  status VARCHAR(20) NOT NULL,
  points INT NOT NULL,
  is_long_term BOOLEAN NOT NULL DEFAULT FALSE,
  goal_id BIGINT,
  time_slot VARCHAR(20),
  display_order INT,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  completed_at DATETIME
);
```

---

### 10.2 reward 表

```sql
CREATE TABLE reward (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100) NOT NULL,
  description VARCHAR(255),
  cost_points INT NOT NULL,
  status VARCHAR(20) NOT NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);
```

---

### 10.3 goal 表

```sql
CREATE TABLE goal (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100) NOT NULL,
  description VARCHAR(255),
  progress INT NOT NULL,
  status VARCHAR(20) NOT NULL,
  start_date DATE,
  end_date DATE,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);
```

---

### 10.4 user_stats 表

```sql
CREATE TABLE user_stats (
  user_id BIGINT PRIMARY KEY,
  total_points INT NOT NULL,
  total_completed_tasks INT NOT NULL DEFAULT 0,
  updated_at DATETIME NOT NULL
);
```

## 十一、推荐 Java 实体拆分思路

如果后端用 Java，建议至少区分：

### Entity

用于数据库映射，例如：

- `TaskEntity`
- `RewardEntity`
- `GoalEntity`
- `UserStatsEntity`

### DTO

用于接口请求和响应，例如：

- `CreateTaskRequest`
- `TaskResponse`
- `CreateRewardRequest`
- `GoalResponse`

### Enum

建议把枚举单独定义，例如：

- `TaskType`
- `TaskStatus`
- `RewardStatus`
- `GoalStatus`
- `TimeSlot`

这样项目会更清晰，也更适合面试讲解。

---

## 十二、第一版建议先固定的规则

为了避免后面改来改去，建议先明确这些规则：

### 1. 任务完成规则

- 每个任务完成一次，只加一次积分
- 已完成任务不能重复加分

### 2. 奖励兑换规则

- 积分足够才允许兑换
- 兑换后立即扣除积分

### 3. 目标进度规则

- 第一版先允许手动设置进度
- 或者只做简单百分比展示

### 4. 长期任务规则

- 长期任务本质仍然是任务
- 只是额外打上 `isLongTerm = true`
- 如有目标关联，则写入 `goalId`

## 十三、第一版建议暂不引入的复杂结构

以下结构建议第二版再加：

- 子任务 `SubTask`
- 任务完成记录 `TaskCompleteRecord`
- 奖励兑换记录 `RewardRedeemRecord`
- 用户表 `User`
- 连续打卡记录
- 目标拆解树
- 任务自动流转记录

第一版先保持轻量，否则你后面会很容易被数据关系拖住。

## 十五、补充建议：如果你想更适合 Java 后端

如果你后面确定要用 Java + Spring Boot，建议你在真正建表前，再做一次“面向数据库”的收敛。

原因是：

- TypeScript 类型写起来方便，但数据库设计更关注字段约束
- Java 项目还要考虑实体类、枚举、请求参数校验、数据库字段命名
- 有些前端友好的字段，到数据库里不一定是最佳方案

例如：

- `status`、`type` 这类字段可以保留字符串枚举
- `progress` 建议用 `INT`，范围明确是 0 到 100
- `points`、`costPoints` 一律非负整数
- 时间字段建议统一命名：
  - `createdAt` / `updatedAt` / `completedAt`
  - 数据库中对应：
    - `created_at`
    - `updated_at`
    - `completed_at`

## 十六、当前版本总结

这份数据结构文档的定位，是给你作为 **MVP 第一版的基础设计稿**。

它已经能支撑：

- 首页任务展示
- 任务完成加积分
- 奖励兑换

而且复杂度还没有高到把你拖住。

你现在最适合做的事情不是继续加字段，而是先做两步：

1. 你自己先按这份文档删一版
   - 把你觉得第一版不会用到的字段划掉
   - 把你觉得必须补上的字段标出来

2. 然后基于“删减后的字段版”去做下一份文档：
   - 接口设计文档
   - 数据库最终建表文档
